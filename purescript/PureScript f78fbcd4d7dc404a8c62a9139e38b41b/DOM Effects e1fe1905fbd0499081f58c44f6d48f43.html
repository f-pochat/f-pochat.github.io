TheÂ mainÂ function uses the DOM and HTML APIs to render our address book component within theÂ containerÂ element we defined inÂ index.html:

  log "Rendering address book component"
  -- Get window object
  w <- window
  -- Get window's HTML document

  -- Get "container" element in HTML

    Just c -> do

      -- Render AddressBook JSX node in DOM "container" element

Can be consolidated to:

Or consolidated even further to:

-- or, equivalently:

It is a matter of personal preference whether the intermediateÂ wÂ andÂ docÂ variables aid in readability.

We trackÂ personÂ as a piece of state with theÂ useStateÂ hook.

Note that you are free to break-up component state into multiple pieces of state with multiple calls toÂ useState. For example, we could rewrite this app to use a separate piece of state for each record field ofÂ Person, but that happens to result in a slightly less convenient architecture in this case.

In other examples, you may encounter theÂ /\Â infix operator forÂ Tuple. This is equivalent to the above line:

useStateÂ takes a default initial value and returns the current value and a way to update the value. We can check the type ofÂ useStateÂ to gain more insight of the typesÂ personÂ andÂ setPerson:

We can strip theÂ Hook (UseState state)Â wrapper off of the return value becauseÂ useStateÂ is called within anÂ R.doÂ block. We'll elaborate onÂ R.doÂ later.

So now we can observe the following signatures:

The specific type ofÂ stateÂ is determined by our initial default value.Â PersonÂ RecordÂ in this case because that is the type ofÂ examplePerson.

Recall thatÂ useStateÂ is used within anÂ R.doÂ block.Â R.doÂ is a special react hooks variant ofÂ do. TheÂ R.Â prefix "qualifies" this as coming fromÂ React.Basic.Hooks, and means we use their hooks-compatible version ofÂ bindÂ in theÂ R.doÂ block. This is known as a "qualified do". Another possible state management strategy is withÂ useReducer, but that is outside the scope of this chapter.

Here we produceÂ JSXÂ which represents the intended state of the DOM. This JSX is typically created by applying functions corresponding to HTML tags (e.g.Â div,Â form,Â h3,Â li,Â ul,Â label,Â input) which create single HTML elements. These HTML elements are actually React components themselves, converted to JSX. There are usually three variants of each of these functions:

  • div_: Accepts an array of child elements. Uses default attributes.
  • div: Accepts aÂ RecordÂ of attributes. An array of child elements may be passed to theÂ childrenÂ field of this record.
  • div': Same asÂ div, but returns theÂ ReactComponentÂ before conversion toÂ JSX.

A second helper function isÂ formField, which creates a text input for a single form field:

Putting theÂ inputÂ and displayÂ textÂ in aÂ labelÂ aids in accessibility for screen readers.

TheÂ onChangeÂ attribute allows us to describe how to respond to user input. We use theÂ handlerÂ function, which has the following type:

For the first argument toÂ handlerÂ we useÂ targetValue, which provides the value of the text within the HTMLÂ inputÂ element. It matches the signature expected byÂ handlerÂ where the type variableÂ aÂ in this case isÂ Maybe String:

In JavaScript, theÂ inputÂ element'sÂ onChangeÂ event is actually accompanied by aÂ StringÂ value, but since strings in JavaScript can be null,Â MaybeÂ is used for safety.

The second argument toÂ handler,Â (a -> Effect Unit), must therefore have this signature:

It is a function that describes how to convert thisÂ Maybe StringÂ value into our desired effect. We define a customÂ handleValueÂ function for this purpose and pass it toÂ handlerÂ as follows:

setValueÂ is the function we provided to eachÂ formFieldÂ call that takes a string and makes the appropriate record-update call to theÂ setPersonÂ hook.

Note thatÂ handleValueÂ can be substituted as:
